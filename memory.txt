# Engine / backtest follow-up (2026-04-18)

## What we measured
Five Python strategies called POST /v1/evaluate with rolling ~1000-bar windows + config_overrides, then simulated stop/target on subsequent bars. Numbers you saw: baseline ~40% win rate, PF ~1.14; looser configs → more trades, worse expectancy.

## Main conclusion: not “broken”, ~zero edge at default R:R
- Default plan: stop = trigger − 2×ATR, target = trigger + 3×ATR (see build_position_plan).
- Breakeven win rate for binary +1.5R / −1R outcomes is 1/(1+1.5) = 40%.
- Baseline 40.1% win rate is exactly on that knife-edge; small positive total R/PF is noise vs fees/slippage and fill assumptions.

## Why relaxing gates hurt (strategies 2–5)
- failed_acceptance still blocks most bars when it fires; other gates were never the only bottleneck for long-run edge.
- Lower vol floor / shorter lookbacks / wider targets increase sample size but admit worse trade population → lower win%, negative avg R (especially 3: wider target needs higher win% — math demands ~27%+ for 1.5 stop / 4 target; you saw ~25%).

## Real code issues / mismatches (worth fixing or documenting)
1) no_runway uses stop_atr_multiple to require clearance, but profit target uses target_atr_multiple — resistance can sit between 2× and 3× ATR and still pass the gate (src/strategies/default/gates/no_runway.rs).
2) Stateless evaluate: each request builds a new StrategyEngine; failed_acceptance is replayed only over failed_acceptance_lookback_bars inside that window (src/machine.rs). Harness must not assume cross-request latch unless runtime_state is extended.
3) Backtest harness risks: intra-bar stop vs target ordering, entry fill model (stop above high), overlapping signals if not deduped — any of these can bias R.

## If goal is >60% win rate
Not compatible with default 2:3 ATR R:R without cutting winners short or asymmetric exits; need different product spec (partial exits, trail, mean-reversion TP, etc.) or accept lower win% with positive expectancy at portfolio level.

# HTTP API: indicators / strategies (2026-04-19)

## What you were trying to achieve
- **Discovery split:** clearer REST than only `GET /v1/catalog` — e.g. dedicated **`/indicators`** (list all indicator leaf paths + warmup metadata) and **`/strategies`** (list all strategy ids), not only one combined catalog blob.
- **One indicator, read:** something like **`GET /v1/indicators/{name}`** (you wrote `mdma` in chat — intent: **one catalog path**) returning exactly what that indicator exposes (schema / `min_bars_required` / notes, and/or last-bar value if paired with data).
- **One indicator, write (test harness):** **`POST`** with the same candle **`MachineRequest`** body as evaluate, but scoped to **a single indicator** — URL carrying the indicator id so you can **probe your payload** against one leaf without running full **`evaluate_multi`**.
- **URL shape you asked for last:** **`POST /indicators/{indicator_name}`** (versioned here as **`POST /v1/indicators/{indicator_name}`**).

## What was actually implemented (this pass)
- **`POST /v1/indicators/{indicator_name}`** wired in `src/bin/server.rs` under the same **`EVALUATE_API_KEY`** + optional **`EVALUATE_MAX_INFLIGHT`** stack as `evaluate` / `evaluate_multi`.
- **Library:** `DecisionMachine::evaluate_indicator` in `src/machine.rs` — builds dataset from `MachineRequest`, takes **last** bar, flattens `PreparedCandle` JSON like catalog/`evaluate_multi`, looks up **`indicator_name` as exact dot-path** (must match `GET /v1/catalog` → `indicators[].path`). Returns **`IndicatorEvaluateResponse`** (`path` + flattened **`IndicatorValueReport`**: `value`, `computable`, `min_bars_required`, `bars_available`, `path_note`). **`EvaluateIndicatorError::Unknown`** → HTTP **404** `{"error":"unknown_indicator","path":"..."}`; prep/serde issues → **400** `invalid_request`.
- **Exports:** `IndicatorEvaluateResponse`, `EvaluateIndicatorError` from crate root (`lib.rs`).
- **Tests:** unit tests in `machine.rs` (happy path `ema_fast_15m`, unknown path). Manual curl: 200 on `ema_fast_15m` + 96 bars, 404 on bogus name.
- **Docs:** README HTTP table row; `context/state.md` bullet.

- **`GET /v1/indicators/{name}`** for metadata-only (no POST) — today you can still read that from catalog + warmup helpers, or GET catalog once and filter client-side.
